# Fluid 产品需求文档（v2 · 对齐版）

> 本版基于 2026-06-14 设计对齐重写。术语见 `CONTEXT.md`,关键权衡见 `docs/adr/`,与 v1 的差异见 `docs/需求文档修订清单.md`。

## 1. 产品定位与核心原则

**Fluid** 是一款面向零代码基础用户的只读代码理解环境,以**自研网页 IDE**(CodeMirror 6)形态交付。其唯一使命是在不修改源代码任何字节的前提下,通过 LLM 生成人类可读的语义投影,帮助用户理解代码的含义。

**核心四律:**

1. **零字节污染**:LLM 生成的所有注释、摘要与解释只存在于内存态的幽灵缓冲区与旁路缓存文件,绝不回写源码。其他 Agent 读取源码的 Token 成本为零。
2. **双粒度语义**:**函数胶囊**为聚合解释单元,**重点行**为最小解释单元。重点行由 AST 启发式判定;非重点行可由用户手动"解释这一行"按需补。
3. **文件级按需激活**:零 Token 的粒度在**文件层**。未打开的文件是绝对真空(零数据、零调度、零渲染、零 Token);**一旦打开文件,即对该文件整体生成函数胶囊与重点行注释**。文件内不再有更细的按需激活。
4. **可信赖的成本契约**:生成产物落盘旁路缓存(键=文件内容 hash + 模型版本),重开命中则零 Token 秒显。追问以当前文件为上下文,继承文件级语义;诚实计费,不承诺"零增量传输"。

## 2. 系统架构

| 模块 | 职责 | 关键产出 |
|---|---|---|
| **上下文图谱** | 管理项目骨架索引(文件树 + understand-anything 的 nodes/edges)与文件级语义实例化 | 两级实例化(L0 骨架 → 文件激活全量) |
| **解释引擎** | AST 解析、函数清单枚举、重点行判定、按函数 LLM 调度与语义投影编译 | 函数胶囊、重点行注释、追问管道 |
| **旁路存储** | 内存态幽灵注释的渲染缓冲 + 落盘缓存 | 玻璃态幽灵注释卡片、函数胶囊、追问条、缓存文件 |
| **前端渲染管线** | 将语义投影绘制到网页 IDE | 流式显影动画、材质一致性、事件上报 |

### 2.1 部署拓扑

**本地后端(Rust)+ 浏览器前端**(形如 `fluid ./project`,见 ADR-0010、ADR-0013):

- **本地后端进程**:读项目源码与 `.understand-anything/`;持有 LLM API key 并做 **LLM 代理**(key 绝不进前端);把旁路缓存写进项目内 `.fluid/`;按需向前端推送生成结果。
- **浏览器前端**:CodeMirror 6 渲染 + tree-sitter WASM 解析(roster/重点行)+ 流式显影。
- **出网边界**:仅 LLM 生成/追问经后端代理出网;解析、骨架索引、缓存命中全本地完成。

## 3. 数据模型

### 3.1 项目骨架输入（understand-anything)

系统读取 `.understand-anything/knowledge-graph.json`。**实测现实(以此为准,而非理想样例):**

- 节点类型:`file` / `class` / `function` / `document`,各带 `summary`、`tags`、`complexity`;class/function 带 `lineRange`。
- **函数节点稀疏**:不覆盖全部函数(实测 33 个 file 节点仅 14 个 function 节点),**无任何行级数据**。
- **edges 丰富**:`calls` / `imports` / `inherits` / `depends_on` / `contains` 等,是模块级上下文的主料。
- **`gitCommitHash` 为空**:不能作缓存键。
- **编码**:`knowledge-graph.json` 实测为**合法 UTF-8**,中文 summary 正确(S2 逐码点核验)。摄入时 UTF-8 直读即可;`encoding_rs` 的 GBK 解码仅作"字节非法 UTF-8"时的兜底,**不对合法 UTF-8 转码**。

因此 understand-anything 是**输入源而非完整真相源**:完整函数清单、行范围、行级注释由 Fluid 自解析 + LLM 补。

**understand-anything 是可选增强,非硬前置**(见 ADR-0011):

- **有 `.understand-anything/`**:用其 file/function summary 与 edges 加速 meso 上下文与跨文件摘要。
- **无**:tree-sitter 出 roster 并解析 import/call 关系替代 edges;文件/函数 summary 由 LLM 生成;跨文件被调对象现生成一句话摘要兜底(可缓存)。Fluid 对任意项目自包含可用。

### 3.2 双粒度结构

- **函数胶囊**:聚合节点。含函数签名、职责摘要、复杂度、输入输出抽象。文件打开时全量生成。
- **重点行**:叶子节点。对应 AST 判定的语义负载行,承载行级幽灵注释。

**关系约束**:重点行挂载于且仅挂载于一个函数胶囊;函数胶囊挂载于且仅挂载于一个文件。

### 3.3 函数清单（roster)

Fluid 自带 AST 解析器枚举每个文件的完整函数清单(名称 + 行范围),并判定重点行。第一阶段支持 **Python 与 Rust**。

- **解析器 = tree-sitter**:一套 grammar 体系覆盖多语言,容错 + 增量解析;新增语言 = 加 grammar。重点行判定表达为对语法树的 tree-sitter **query**(每语言一份)。
- **解析位置 = 浏览器端(WASM)**:零网络往返、零 token、源码不必上传即出 roster 与重点行。仅 LLM 生成步骤出网。
- 详见 ADR-0009。

## 4. 两级激活链与生命周期

### 4.1 两级实例化状态

| 级别 | 触发 | 内存内容 | 说明 |
|---|---|---|---|
| **L0 骨架** | 引擎冷启动 | 文件树 + understand-anything 的 nodes/edges(O(nodes) < 100KB) | 零 LLM,仅目录与节点索引 |
| **文件激活** | `FileActivated` 事件 | 该文件函数胶囊 + 重点行注释(按函数流式填充;大文件视口渐进,见 §4.4) | 命中旁路缓存则零 Token 秒显,否则按函数调 LLM |

> v1 的 L1/L2 拆分与"展开函数才出行级""视口±5 闸门"已废弃(见 ADR-0002)。

### 4.2 闸门规则

- **文件级闸门**:未激活文件的所有查询统一返回 `INACTIVE_CONTEXT`,阻断 LLM 调度。这是唯一的硬闸门。
- 文件激活后,该文件内函数胶囊与重点行注释一律生成,无更细闸门。

### 4.3 生成形态

- 打开文件时,生成请求**按函数拆分**:每个请求 = 一个函数胶囊 + 该函数的重点行注释,逐个流式显影。
- 每个请求注入**共享上下文**,至少包含:文件级 summary、**本文件函数清单 roster**、相关 edges(calls/imports)。
- 详见 ADR-0004。

### 4.4 生成调度与大文件降级

- **调度**:按**视口邻近度**排序(可见区函数最先,向上下扩散),**有限并发**(同时 3-5 个在途,受 API 速率/成本约束);用户滚动即重排优先级。
- **小文件**(阈值内,如 ≤20 函数且 ≤800 行):打开即**全量 eager 生成**,顺序仍服务于视口。
- **大文件**(超阈值):退回**视口渐进生成**——只生成视口邻近批,其余函数显示"滚动到此处自动生成"占位,进入视口时按调度生成。已生成函数的折叠/展开仍是纯 UI 零重算。
- 阈值具体数值实现期定。详见 ADR-0008。

### 4.4 内存与显隐

- **函数折叠/展开**:纯前端 UI 显隐,**不触发生成或销毁**。
- **文件关闭(FileClosed)**:释放该文件内存态,系统回到 L0;生成产物已落盘旁路缓存,重开命中。

## 5. 缓存策略

### 5.1 旁路缓存（落盘）

- **键**:`文件内容 hash + 模型版本`(绕开为空的 gitCommitHash)。
- **内容**:已生成的函数胶囊与重点行注释。
- **位置**:源码之外的旁路文件(候选 `.understand-anything/fluid-cache/` 或 `.fluid/`,实现期定)。
- **失效**:文件内容变(hash 变)或模型/prompt 版本变 → 失效重算。命中则零 Token 秒显。
- 详见 ADR-0003。

### 5.2 文件级会话上下文（内存)

- 文件激活时构建,含项目级(macro)、文件/模块级(meso,含 edges)、当前文件函数摘要(micro)。
- 追问以此为上下文;文件关闭时释放。

## 6. LLM 调度与追问器

### 6.1 调度闸门

LLM 调度器只受**文件是否激活**这一层硬闸门控制:未激活则休眠,激活则按函数生成。手动"解释这一行"为用户主动触发的单行补充请求。

### 6.2 追问管道（分层上下文)

追问器是当前激活文件的增量查询终端,上下文以**当前文件**为边界,**分层装配**以免大文件爆窗(见 ADR-0006):

- **默认层**:文件级 summary + **全部函数胶囊的摘要**(便宜、已生成) + edges + **focus 指向的函数/行的完整源码与注释**。整个文件以摘要粒度在场(全局视野),聚焦处以源码粒度在场(精确)。
- **降级层**(摘要总和仍超窗):退到 focus 邻近函数的源码/摘要,远端函数仅留名称。
- **回退**:模型可请求"追加某函数源码"→ 系统补该函数源码再答一轮。

### 6.3 跨文件上下文规则

- **默认**:函数 A 通过 `calls`/`imports` 指向的跨文件目标,仅注入其**一句话摘要**,**绝不拉源码、绝不打开目标文件**。摘要来源优先级:understand-anything 节点 summary(近乎免费) → 若无,则对该被调对象**现生成一句话摘要并缓存**(ADR-0011) → 仍不可得则留函数名。
- **临时跨文件取源(Ephemeral Cross-File Fetch)**:仅当用户明确追问"未打开文件里某被调函数的内部实现"时触发。系统只取**该函数的源码片段**(按 lineRange/AST),用于本轮回答后**立即丢弃**。
  - 护栏:范围 = 单个函数 span,非整文件;不写目标文件的旁路缓存;不计为 `FileActivated`,目标文件事后仍是真空;仅在用户对该被调对象的实现有明确意图时触发。
  - 详见 ADR-0007。

**接口规范:**

```typescript
interface queryInline {
  sessionId: string;
  filePath: string;          // 上下文 = 整个该文件
  focus?: {                  // 可选:聚焦到某函数/行,仅用于提示与排序
    type: 'function' | 'line';
    id: string;
  };
  userQuery: string;
  previousTurnId?: string;   // 同文件连续追问续接
}
```

> 注:无状态 LLM API 每次调用必须重发上下文。v1 的"仅传 userQuery / 请求体不含上下文"已删除——物理不可达。成本以"计费 token 增量"衡量;同文件连续追问通过保持文件级上下文 + prompt 缓存降低成本。

### 6.3 追问器 UI 规范

- **位置**:右下角常驻折叠浮窗。
- **折叠态**:32px,仅露输入框顶条。
- **展开态**:自适应,最大 480px。
- **材质**:与幽灵注释层一致的玻璃态——`backdrop-filter: blur(8px)`、背景 92% 不透明、1px `#333` 边框、`Inter` 13px。输入框底色 `#1f1f1f`。
- **锚点提示**:顶部 11px `#8b949e` 灰字静默显示当前焦点,如 `基于 execute() · 函数级` / `基于 execute() · 第 15 行`。
- **增量动画**:回复卡片从追问条上方滑出(`translateY(8px→0)`、`opacity(0→1)`,200ms,ease-out)。禁止骨架屏/闪烁。
- **文件切换过渡**:一次 <150ms 的环境色温过渡,暗示上下文根系切换。

## 7. 旁路存储与视觉显影规范

> **适用范围(ADR-0015,2026-06-15)**:整体界面采用**类 VSCode IDE 壳**(活动栏 / 资源管理器 / 文件 tab / 面包屑 / 可拖宽面板 / Open Folder / 状态栏 / 命令面板),走标准 IDE 视觉。本节(§7.1–§7.5)的显影动画与**§7.4 玻璃态材质连续性铁律,仅约束代码区内的幽灵注释**(函数胶囊 + 行级注释),**不约束外层壳**。

### 7.1 真空态（未打开文件)

- gutter 无任何指示点,代码栏纯粹无旁路痕迹。
- 追问器休眠,输入框置灰:"打开文件以启用追问"。

### 7.2 函数级显影（文件激活后,按函数流式凝结)

函数胶囊作为语义凝结核优先逐个显影:

1. 卡片高度 `0 → contentHeight`(300ms,`cubic-bezier(0.25,1,0.5,1)`)。
2. 文字 `opacity 0 → 1`(200ms,delay 100ms)。
3. 左侧彩色竖线 `scaleY(0→1)`,origin top(350ms)。

gutter:函数顶部 2px 空心微点(stroke `#484f58`)。

### 7.3 行级显影（重点行,随所属函数胶囊一同填充)

1. 重点行幽灵注释卡片高度 `0 → auto`(250ms)。
2. 文字 `opacity 0 → 1`(200ms,delay 50ms)。
3. 左侧语义色温竖线生长(300ms)。

gutter:重点行 gutter 指示渐变为实心脉动点(6px 圆环,`opacity 0.4→0`,800ms 循环)。非重点行无指示点;用户 hover 可触发"解释这一行"。

### 7.4 材质连续性铁律

- 函数胶囊与重点行注释必须使用同一套玻璃态渲染管线。
- 禁止圆角/阴影/字体差异。用户从函数级扫视至行级,感知为同一介质的不同深度。

### 7.5 折叠态

函数胶囊折叠后收缩为 4px 色温边线(记忆锚点);内部行级注释**仅隐藏不销毁**(已在内存/缓存,展开即现,零重算)。

## 8. 接口契约与数据层安全

### 8.1 对外接口闸门

| 接口 | 输入 | 正常返回 | 闸门拦截返回 |
|---|---|---|---|
| `resolveFileContext` | `filePath` | 文件级摘要 + 函数胶囊列表 + 重点行注释 | `INACTIVE_CONTEXT`(文件未激活) |
| `explainLine` | `filePath, functionId, lineNumber` | 单行幽灵注释 | `INACTIVE_CONTEXT`(文件未激活) |
| `queryInline` | `sessionId, filePath, focus?, userQuery` | `{ answer, confidence }` | `INACTIVE_CONTEXT`(文件未激活) |

> 文件激活后,函数/行级接口不再独立设防——文件是唯一闸门。`explainLine` 为长尾非重点行的手动补充。

### 8.2 INACTIVE_CONTEXT 占位对象

```json
{
  "status": "INACTIVE_CONTEXT",
  "summary": "当前文件尚未打开,仅提供骨架信息。",
  "skeletonOnly": true
}
```

## 9. 性能度量与验收标准

| 指标 | 目标值 | 验证方法 |
|---|---|---|
| 未打开文件内存增量 | < 100KB(仅骨架) | 冷启动前后堆快照对比 |
| 缓存命中重开文件首屏 | < 200ms | 命中旁路缓存时,FileActivated 到首胶囊渲染 |
| 首次生成·首个函数胶囊出现 | 流式,目标 < 首个 LLM 响应时延 | 埋点:FileActivated 到第一个胶囊 `opacity=1` |
| 首次生成·整文件显影完成 | 随函数数量线性,流式无阻塞 | 埋点:到最后一个函数胶囊填充完成 |
| 未打开文件 Token 消耗 | 0 | 对未打开文件调用任意接口,断言 `INACTIVE_CONTEXT` 且无 LLM 调用 |
| 缓存命中 Token 消耗 | 0 | 内容/模型不变时重开文件,断言无 LLM 调用 |
| 追问计费增量 | 仅新增 turn 的 token | 统计同文件第二次追问的计费 token(含 prompt 缓存折扣) |
| 函数折叠/展开 | 0 重算 | 折叠再展开,断言无 LLM 调用 |

> v1 的"<500ms 行级显影""抓包请求体不含上下文"已删除(LLM 物理限制,见修订清单)。

------

## 10. 一句话交付标准

用户未打开的文件在系统中绝对不存在;打开后,该文件的函数胶囊与重点行注释按函数逐个凝结显影,折叠展开零重算,重开命中缓存零 Token 秒显;右下角追问条以整个当前文件为上下文随时应答。整个过程中,源码层零字节未被改写。
