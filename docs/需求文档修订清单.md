## 数据现实修正(基于真实 understand-anything 产物)

- **§3.1 gitCommitHash 为空**:实测 `meta.json` 与 `project.gitCommitHash` 均为 `""`,不能作缓存键 → 改用文件内容 hash。
- **函数节点稀疏**:实测 33 个 file 节点仅 14 个 function 节点、40 个 class 节点(共 91 节点),**不覆盖全部函数**;**零行级数据**。故完整函数清单与行级注释必须由 Fluid 自解析 + LLM 补(ADR-0004, 0005)。
- **edges 是宝藏**:实测 199 条 edges(calls 43 / imports 44 / inherits / depends_on ...),需求文档未提,应纳入函数胶囊的共享上下文。
- **编码(S2 已核实修正)**:understand-anything 输出 `outputLanguage: zh`,JSON 实测为**合法 UTF-8、中文 summary 正确**(逐码点验证 `U+6267 U+884C`=执行)。早期"GBK 乱码"的印象是 Git Bash 控制台无法渲染中文的显示假象,**非数据损坏**。Fluid UTF-8 直读;`encoding_rs` GBK 仅作"字节非法 UTF-8"的兜底,绝不对合法 UTF-8 转码(否则反而损坏)。

## 本次新增/确认的决策(不在原文中)

- 重点行由 **AST 启发式**判定;长尾非重点行由用户**手动"解释这一行"**按需补。
- 生成**按函数拆分、逐个流式显影**,共享上下文至少含"本文件函数清单"。
- 第一阶段支持 **Python + Rust**,Fluid 自带 AST 解析器。
- 交付载体 = **自研网页 IDE**(Monaco/CodeMirror 二选一待定)。
