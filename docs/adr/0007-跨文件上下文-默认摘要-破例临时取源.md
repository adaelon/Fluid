# 跨文件上下文:默认只注入摘要,破例临时取源

代码理解高频需要"这行调用的函数是干嘛的",而被调函数常在别的文件。默认做法:对 `calls`/`imports` 指向的跨文件目标,只注入其 understand-anything 节点的一句话 summary(L0 骨架已有,近乎免费),不拉源码、不打开目标文件——既能答出"调的是做加密签名的 B",又不破文件级零 Token。

## Considered Options

- **纯文件封闭(被调函数只留名)**:答案干瘪,无法说明跨文件调用的作用。
- **跨文件全量拉取源码**:破坏文件边界,成本与窗口都失控。

## Consequences

- 被调函数不在 understand-anything 节点中(节点稀疏)时,只能留函数名,可接受。
- **破例:临时跨文件取源(Ephemeral Cross-File Fetch)**——当用户明确追问未打开文件里某被调函数的内部实现时,只取该函数的源码片段,用完即弃。护栏:单函数 span 而非整文件;不写目标文件旁路缓存;不计为 FileActivated,目标文件事后仍真空;仅在用户对该实现有明确意图时触发。这是对核心律 3 的有意、有界、用户触发的破例,设此护栏防止腐化为"打开一切"。

## 修订(2026-06-15,S10c)

**"明确意图"的实现形态 = 模型在用户追问语境下点名**。S10c 落地此破例,触发路径复用 S10a-追源 的两段式规划(ADR-0017):用户就当前文件追问 → 降级规划阶段把"当前文件调用、定义在别处且 graph 可定位"的跨文件 callee 也列入 fetchable → 模型点名 `{"need":[...]}` → 后端经 graph 节点(file_path + line_range)定位 → `reader.read_file` + `slice_span` 切出该函数源码注入,phase-2 重答。

- **"用户触发"软化为"用户追问语境 + 模型点名"**:触发权名义在模型,但锚在用户的具体追问(模型不会无追问自发取源),且**护栏全保留**:仅单函数 span、`reader.read_file` 纯读不写 `.fluid` 缓存、不激活目标文件(事后仍真空)、有字符上界(与同文件追源共享 `QUERY_FETCH_BUDGET_CHARS`)。"防止打开一切"由"模型只点名 graph 已索引的少数 callee + 总量上界"保证,而非由"必须用户手点"保证。
- **否决"前端 callee chip 点击"**(真·用户手点):最贴字面,但需后端先暴露 callee 清单 + 前端新交互,工作量大;模型点名在用户追问语境下已足够表达意图。
- **与同文件「按需追源」名池不相交**:跨文件 fetchable 排除当前文件 roster 里已有的名(本地优先),避免模型点名歧义。
