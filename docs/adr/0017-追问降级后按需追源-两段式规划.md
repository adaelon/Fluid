# 追问降级后的按需追源:两段式规划调用,仅当降级时启用

追问上下文超窗降级后(S10a-降级,远端同文件函数被降为**仅留名、无摘要无源码**),涉及这些函数实现细节的追问可能答不准。回退 = **两段式 round-trip**:第一段非流式「规划调用」给模型当前文件上下文 + 仅留名函数的名单,问它「要哪些函数的源码才能答准」→ 出 JSON `{"need":[...]}`;后端按名切出这些函数源码(同文件、有上界)追加进上下文;第二段在加料后的上下文上**流式作答**。**门控**:只在降级真正发生时启用(常规文件维持单次流式调用);**一轮不递归**(规划→作答各一次,防自循环)。

## Considered Options

- **Tool-use / function calling**:定义 `fetch_function_source` 工具,流式中模型调用。语义最干净,但依赖 opencode zen 网关 + glm-5.1 的流式 tool-call 支持(未验证),且 SSE 解码要兼顾 tool-call 分帧,复杂度高。
- **流内哨兵标记**:模型需源码时吐 `<<FETCH:name>>`,后端流中检测→补源→续答。污染答案流、需跨 chunk 哨兵配对容错,脆。
- **总是两段式(不门控)**:覆盖面最广(未降级也能追源),但每次追问都 2 次 LLM 调用,成本/延迟翻倍。

## Consequences

- 降级文件的一次追问 = 2 次 LLM 调用(规划 + 作答);常规(未降级)文件维持 1 次。
- 规划输出是模型选的**函数名单(输入)**,真实源码由后端确定性切片产出 —— 无 AI 自评闭环(B2)。规划失败/坏 JSON/点名为空 → 退回降级答案(不加料),整次追问绝不因追源而失败。
- 追源**仅限同文件**且有上界(`QUERY_FETCH_BUDGET_CHARS`),phase-2 仍有界;跨文件取源仍是独立的 ADR-0007 破例(S10c)。
- 需追问请求携带函数行范围(`rosterSpans`)以便后端按名切源;老客户端不带 → 优雅跳过追源(降级答案)。
- 规划点名了不可追源/不存在的函数 → 忽略(模型幻觉护栏)。
