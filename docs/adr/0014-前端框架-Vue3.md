# 前端框架定为 Vue 3

前端在 Vanilla TS / React / Vue 三者中选 **Vue 3 + TypeScript**(SFC + Vite)。判断基准:Fluid 的视觉重头(行间玻璃卡片、gutter 脉动点、流式显影)全在 CodeMirror 6 内部,走 CM6 自己的 decoration/widget **命令式** API——这部分框架无关,任何框架都得用生命周期钩子 + `shallowRef`/`markRaw` 包一层命令式逃生舱手动持有 `EditorView`。框架只在 CM6 之外的那层 DOM(文件树侧栏、追问器浮窗、S10 问答卡片增量列表)兑现价值。在此前提下选 Vue:① Vite 是 Vue 团队出品,搭配最原生、构建/HMR 摩擦最小;② SFC 的 `<style scoped>` 让 §7 玻璃态材质就近写且作用域隔离,利于 S11 视觉打磨;③ 侧栏/浮窗的有状态 DOM 用 `ref`/`v-for` 省样板,与 React 同档。

## Considered Options

- **Vanilla TS(无框架)**:最贴 `技术方案.md §1` 原技术栈、零额外依赖、与 CM6 命令式心智同源;但文件树展开态、追问器问答卡片增量列表须手写 DOM 更新,样板多、状态同步易漏。本项目非 CM6 的 UI 虽少,长期(S10 追问器)仍偏脆。
- **React + TS**:生态最大、切片计划 `.tsx` 命名本就为它;但相对 Vue 在本项目无额外优势——同样要 `useEffect` 逃生舱包 CM6,同样要改架构文档,Vite 搭配不如 Vue 原生,§7 样式就近性不如 SFC `<style scoped>`。

## Consequences

- **逃生舱税(固定成本)**:CM6 的 `EditorView`、GhostStore 等命令式大对象必须 `shallowRef`/`markRaw`,**严禁进普通 `ref`**——否则 Vue 响应式深度代理 CM6 内部状态会损坏编辑器。这是 Vue 集成命令式库的硬约束,须在代码注释与评审中守住。
- **技术栈与文档**:`技术方案.md §1` 前端栈补登 Vue 3;切片计划组件文件 `.tsx` 命名改为 `.vue`。`web/render/*`、`web/parser/*`、`web/scheduler.ts` 等 CM6/tree-sitter 命令式模块仍是 `.ts`,不进框架。
- **框架只惠及一角**:CM6 与 tree-sitter WASM 两大核心库框架无关,Vue 的模板与响应式在主战场用不上,收益集中在侧栏/浮窗。这是有意接受的——框架为非 CM6 的 UI 引入,不为画布。
- 拓扑(ADR-0010)、编辑器内核(ADR-0012)不变,仅前端框架落定。
