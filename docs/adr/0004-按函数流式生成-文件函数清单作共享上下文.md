# 按函数拆分流式生成,文件函数清单作共享上下文

打开文件时,生成请求**按函数拆分**——每个请求 = 一个函数胶囊 + 该函数的重点行注释——逐个流式显影(贴合需求文档 §7.2「函数胶囊作为语义凝结核优先显影」)。每个请求注入共享上下文,至少包含:文件级 summary(来自 understand-anything)、**该文件完整函数清单 roster**(由 Fluid 自带 AST 解析得出)、相关 edges(calls/imports)。

## Considered Options

- **整文件单次 LLM 调用**:跨函数上下文最全,但大文件输出易截断(几十个胶囊 + 上百行注释挤一个响应),首屏须等整个响应,体验差。

## Consequences

- 输出有界、可并发、首个胶囊更快凝结。
- 轻微的跨函数上下文损失,由共享注入的函数清单 + edges 补偿。
- 函数清单是硬要求:函数胶囊必须知道"本文件里还有哪些函数"。
