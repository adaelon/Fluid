> **跨文件取源 = 扩展两段式规划、模型点名、门控 fetchable 非空、护栏结构化**:build_query_prompt 本层唯一持源是 focus;跨文件函数对模型是盲区。复用 S10a-追源 round-trip,把 graph 可定位的跨文件 callee 也列入 fetchable → 模型点名 → 后端经 graph 节点(file_path+line_range)定位、`reader.read_file`+`slice_span` 切源加料。**"用户明确意图"软化为"用户追问语境 + 模型点名"**(ADR-0007 修订):触发权名义在模型但锚在用户具体追问,护栏全保留(单函数 span / 纯读不写 `.fluid` / 不激活 / 共享 `QUERY_FETCH_BUDGET_CHARS` 上界)。门控由「仅降级时」扩为「fetchable 非空即触发」(ADR-0017 修订):双调用成本限定在"有图+有可取跨文件目标"子集,小文件问跨文件实现也生效。否决前端 callee chip(需新交互,工作量大)、否决总是两段式(每问翻倍)。命门:① 锁内预读会读到模型最终未点的候选目标文件(纯读、用完即弃、不缓存不激活,仍 ephemeral;distinct path 去重,数量有界);② 跨文件 callee 受 graph 稀疏性限(节点无 line_range 则不可取,留名);③ 同名跨文件函数(多文件)按名去重取首个,模型无法区分,罕见;④ 一轮不递归(沿 ADR-0017)。决策见 ADR-0007/0017 S10c 修订。

**入口**:`WS /api/query` → 若当前文件有 graph 可定位的跨文件被调函数(或同文件降级函数)→ 规划点名含跨文件名 → 模型点跨文件 callee → 后端从他文件切该函数源、label `name @ path` 注入 → phase-2 流式答(答案可引跨文件实现);目标文件事后仍真空(零字节、未缓存、未激活)。无 graph/无跨文件可取 → 维持单程,行为不变。
