## 2026-06-14 S2 · 图谱加载(可选 + 正确解码)

**触达**:
- `crates/fluid-server/Cargo.toml` — 加依赖 `encoding_rs`(仅 UTF-8 失败时兜底)。
- `crates/fluid-server/src/graph_loader.rs:GraphLoader::load` — 读 `<root>/.understand-anything/knowledge-graph.json`;缺失/损坏→`None` 不崩(ADR-0011)。
- `crates/fluid-server/src/graph_loader.rs:decode_text` — **UTF-8 优先,GBK 仅作非法 UTF-8 兜底**;另剥离 BOM。
- `crates/fluid-server/src/graph_loader.rs:{KnowledgeGraph,GraphNode,GraphEdge}` — 类型(camelCase serde rename,API 直出);node 含可选 `lineRange:[u32;2]`/`languageNotes`。

- `crates/fluid-server/src/main.rs:main` — 注入 `GraphLoader` 到 `AppState`,启动打印节点/边数或"无图谱"。

**入口**:`GET /api/project/graph` → 有 `.understand-anything/` 返回 nodes/edges,无则 `null`(HTTP 200)。

**测试**:
- `cargo test`(`graph_loader::tests`,4 条):解析含中文 summary 的最小图 / UTF-8 直通 / 非法 UTF-8 退 GBK / 缺失图谱→None。全套 9/9 绿。
- 端到端(B2):alphaGPT 解出 **91 节点/199 边**,summary 中文完好(`执行模块的配置类…`),`lineRange [8,38]`、9 类 edge 齐;对无图谱的 Fluid 仓库返回 `null` 且 `/api/project/tree` 仍 200(服务存活)。

> **重大事实修正**:技术方案/ADR-0011 注/修订清单称 summary 是"GBK 错位乱码须 encoding_rs 解码"——实测 `knowledge-graph.json` 是**合法 UTF-8 正确中文**(逐码点验证 `U+6267 U+884C`=执行),所谓乱码是控制台无法渲染中文的显示假象。故对合法 UTF-8 **不做** GBK 转码(否则反而损坏);GBK 降为非法 UTF-8 时的兜底。

## 2026-06-14 S4 · tree-sitter 解析(Python + Rust,WASM)

**触达**(全部新增,纯前端解析层,不接 LLM/渲染 — ADR-0005/0009):
- `web/package.json` — 加依赖 `web-tree-sitter@0.24.7`(**非最新 0.26**,见下)、`tree-sitter-wasms@0.1.13`(预编译 grammar)。
- `web/tsconfig.json` — 加 `esModuleInterop`(`web-tree-sitter` 是 CJS `export =`,默认导入需此项)、`allowImportingTsExtensions`(内部用显式 `.ts` 路径,兼容 Node 24 原生跑 `.ts`)。
- `web/src/parser/types.ts:{ParserLang,FunctionSpan,FileParse}` — 解析产物类型(技术方案 §3);`keyLines: Map<fnId, number[]>`。
- `web/src/parser/parse.ts:FluidParser` — 环境无关核心。`create(assets)` 初始化 WASM + 编译 query;`parse(lang, source)` 出 `{roster, keyLines}`。`extractRoster`(roster query 配对 `@fn`/`@name`,id=`name#startLine`)、`extractKeyLines`(keyline 捕获 → `innermostHost` 归属到最内层函数,类/模块级行丢弃 = 重点行只挂函数)。
- `web/src/parser/keyline-queries/python.scm` / `rust.scm` — 重点行判定 query(每语言一份,PENDING 定稿规则见下)。
- `web/src/parser/index.ts` — 公开面;浏览器 wasm/.scm 资产装配留给 S7。
- `web/scripts/parse-check.ts` — S4 确定性验证脚本(Node 24 原生 strip TS),喂真实 grammar + 真实 `.scm` 跑真实 `FluidParser`。

**PENDING 定稿 — 重点行规则**:行位于**函数体内**且首节点 ∈ {赋值(`assignment`/`let_declaration`/`assignment_expression`)、有值 `return`、`raise`/`assert`、分支/循环/with 头(`if`/`for`/`while`/`match`/`with`)、语句位调用、Rust 块尾调用(隐式返回)}。**跳过**:import/use、装饰器/属性宏、`def`/`class`/`fn`/`impl` 结构头、`pass`、try-except 脚手架头(其内部语句仍算)。类级/模块级行不挂任何函数胶囊 → 非重点行。**v1 已知局限**:Rust 非调用型尾表达式(如 `&self.root`)不算重点行。

**入口**:`FluidParser.create([{lang,grammarWasm,keyLineQuery}])` → `parser.parse('py'|'rs', source)`。当前仅被验证脚本调用;CM6/GhostStore 接线在 S7。

**测试**(确定性 B2 — `web` 无单测框架,验证脚本 + 人工核对):
- `node scripts/parse-check.ts` 对 `alphaGPT/execution/config.py`:3 个 classmethod;重点行 `has_private_key=[22]`、`get_payer_keypair=[26,27,28,29,31,33,34]`、`get_wallet_address=[38]` — 与人工逐行核对**一致**。
- 同脚本对 `crates/fluid-server/src/project_reader.rs`:17 个函数;重点行落在 let/调用/分支/return,平凡 getter(`root`/`path`)为空,空行与结构头正确跳过 — 人工核对**合理一致**。
- `npm run build`(`vue-tsc --noEmit` + `vite build`)**绿**(parser 文件在 `src/**` 内被 vue-tsc 类型检查通过;尚未进 bundle,因未被 UI 引用,符合 S4 范围)。

> **版本钉死 web-tree-sitter@0.24.7(非最新 0.26.9)**:0.26 重写了 Emscripten dylink 加载,`tree-sitter-wasms@0.1.13` 的预编译 grammar 报 `getDylinkMetadata` 失败;钉 0.24.7(最后一个 pre-rewrite)即可直接用预编译 wasm,免装 tree-sitter CLI + emscripten 重编 grammar。**何时回头**:需升级或自编新语言 grammar 时,重评是否上 0.26 + 自建工具链。
